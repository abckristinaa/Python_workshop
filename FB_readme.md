**API Graph**
API Graph — это основной способ чтения и записи данных в социальный граф FB, 
используемый приложениями. Все наши SDK и продукты взаимодействуют с API Graph,
 а остальные наши API являются расширениями API Graph.
 
 The Graph API is HTTP-based, so it works with any language that has an HTTP 
 library, such as cURL and urllib. Все данные передаются по протоколу HTTP/1.1,
  а для работы эндпойнтов необходим HTTPS.
  
Почти все запросы передаются в URL хоста graph.facebook.com. 
Единственным исключением являются загрузки видео, которые используют для этого 
graph-video.facebook.com.

**Маркеры доступа**

С помощью маркеров доступа приложение может взаимодействовать с API Graph. 
Как правило, они выполняют две функции:

    1) предоставляют приложению доступ к информации о пользователе без пароля;
    2) позволяют идентифицировать приложение, его пользователя и тип данных, 
    доступ к которым был предоставлен вашему приложению.

Почти для всех конечных точек API Graph необходим тот или иной маркер доступа, 
поэтому при обращении к конечной точке может потребоваться указание маркера в 
запросе. Требования конечных точек к маркерам доступа можно найти здесь:
https://developers.facebook.com/docs/graph-api/reference

Маркеры доступа работают по протоколу OAuth 2.0.
Как правило, это выполняется в рамках веб-интерфейса. После авторизации 
приложения могут получать доступ к информации с помощью этих маркеров.

    Как мы видим, это интерфейс Facebook. Пользователь вошел в свой аккаунт 
    через этот интерфейс, в результате чего мы можем авторизовать этого человека. 
    Если он продолжит работу в приложении, мы обменяем старый маркер (маркер приложения) 
    на новый (маркер пользователя). Затем, используя новый маркер пользователя, 
    приложение сможет выполнять запросы API Graph, но получит доступ только к фото, 
    видео и электронному адресу именно этого пользователя.

Это важное свойство маркеров доступа. ID приложения и пользователя (помимо 
прочих данных) закодированы в маркере и позволяют нам отследить, к каким данным 
пользователь предоставил доступ приложению.

Поскольку маркеры разрешают доступ к данным человека, а использовать их может 
кто угодно, в запросах с ними нужно быть очень осторожным. **Для этого мы рекомендуем 
обрабатывать токены с помощью «Входа через Facebook».**


**Вход через Facebook**
Вход через Facebook позволяет людям быстро создать аккаунт и войти в ваше 
приложение с различных платформ. 
Вход через Facebook позволяет решать две задачи: выполнять аутентификацию и 
запрашивать разрешения на доступ к данным людей.
Вход через Facebook можно использовать и просто для авторизации, и для 
авторизации и запроса разрешений.

1. Проверка статуса входа

Приложения, использующие наши SDK, могут отслеживать вход в приложение с помощью
 встроенных функций. Во всех других приложениях должен быть собственный механизм, 
 позволяющий запомнить, когда был выполнен вход. Если этот индикатор отсутствует, 
 в приложении будут доступны только функции, не требующие входа. Если человек 
 выполнил выход, приложение в подходящий момент должно перенаправить его в 
 диалог "Вход", например при нажатии кнопки "Вход".
